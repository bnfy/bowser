# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Bowser ‚Äî a minimal Electron browser shell: custom-drawn "Island" chrome (a single floating command pill replacing the traditional tab strip + toolbar, per the Bowser Design System's Island Chrome handoff) with ad/tracker blocking wired in at the network layer, independent of Chrome's extension store and Manifest V3's `declarativeNetRequest` limits. Plus favorites, history, downloads, settings, slash commands, a Quick Switcher, private tabs, an explicit permission policy, and packaging config.

## Commands

```
npm install
npm start                       # run in dev (electron .)
npm run dist                    # installable build: macOS dmg/zip, Windows NSIS, Linux AppImage
npm run dist:dir                # quick unpacked build in dist/, no installer
npm run release                 # scripts/release.sh: clean, build, sign, publish to GitHub Releases
```

There is no test suite and no linter configured in this repo ‚Äî don't assume `npm test` or `npm run lint` exist.

`npm run release` bumps nothing itself ‚Äî bump `version` in `package.json` (and consider the `electron` devDependency, since Chromium can't be swapped out of a running app) *before* running it. It shells out to `scripts/release.sh`, which authenticates via the `gh` CLI's own cached session (no `GH_TOKEN` needed locally), builds unpublished, then uses `gh release create`/`gh release upload` directly instead of electron-builder's own GitHub publisher ‚Äî that publisher races its per-artifact upload tasks against each other on first publish for a tag and can leave a release missing `latest-mac.yml` or a blockmap; shelling out to `gh` sequentially avoids that. The script is safe to re-run: if the release for the current version already exists, it fills in/overwrites assets (`--clobber`) instead of failing. It also wipes `dist/` before building so stale artifacts from earlier versions or renamed builds never linger.

**Notarization:** the build step is wrapped in `op run --env-file=.env.1password --no-masking -- ...`, pulling `APPLE_ID`/`APPLE_APP_SPECIFIC_PASSWORD` from the 1Password item **"Apple Notarization"** (vault Dev, `username`/`password` fields) plus a literal `APPLE_TEAM_ID` ‚Äî electron-builder's `@electron/notarize` integration auto-detects those three env vars with no extra config (hardened runtime + the JIT/unsigned-memory/disable-library-validation entitlements it needs are already electron-builder's defaults). If `op` isn't installed, the script falls back to an unnotarized build with a warning instead of failing outright; if `op` *is* installed but the item can't be resolved (locked vault, wrong item name), it fails loudly rather than silently shipping unnotarized ‚Äî a signed-but-unnotarized build gets quarantined and blocked by Gatekeeper the moment it's downloaded (e.g. by `electron-updater` during an auto-update), which is exactly what broke a stale pre-rebrand install after the v0.3.0 release.

**Windows & Linux:** `release.sh` only builds macOS locally (it's a macOS dev machine), then dispatches `.github/workflows/release-windows-linux.yml` (`gh workflow run ... -f tag=$TAG`) and watches it to completion. That workflow builds NSIS on `windows-latest` and AppImage on `ubuntu-latest` and uploads both onto the same release tag (`--clobber`, so it's re-run-safe like the mac path). `workflow_dispatch` only works off a workflow file already on the default branch, so a brand-new/edited workflow won't be dispatchable until merged ‚Äî `release.sh` warns and exits 0 (without failing the mac release) if the dispatch call fails, printing the manual `gh workflow run` command to run once that's sorted. Windows code signing is opportunistic: the workflow reads `CSC_LINK`/`CSC_KEY_PASSWORD` repo secrets (electron-builder's own auto-detected env vars, an Authenticode cert + password) and signs if they're set, or builds unsigned with a `::warning::` annotation if not ‚Äî an unsigned installer triggers a SmartScreen "unknown publisher" prompt, so set those secrets before pointing real users at the Windows download. Linux AppImages aren't code-signed (no OS-level equivalent to Gatekeeper/SmartScreen to satisfy).

First launch is slower than usual: the ad blocker fetches and compiles EasyList + EasyPrivacy, then caches the compiled engine in the app's userData directory (`adblock-engine.v<N>.bin`) so later launches are instant. Delete that file to force a refresh.

Auto-update (`electron-updater` against GitHub Releases) only runs in packaged builds ‚Äî `npm start` skips it entirely. Releasing means bumping the `electron` devDependency (it tracks Chromium stable) and `version` in `package.json`, since Chromium itself can't be swapped out of a running app.

## Architecture

**One `BrowserWindow`, many `WebContentsView`s.** The window's own `webContents` renders `src/renderer/index.html`/`styles.css` ‚Äî the chrome **strip**: a slim (56px) band holding the resting island pill (tab dots, favicon, domain, shield count, private chip), window controls, and permission prompts. Each tab is a separate `WebContentsView` added as a child of `win.contentView` at `y = chromeHeight`; only the active tab's view is attached, so switching tabs is remove-one/add-another rather than destroying anything. `src/main/main.js` owns all of this: the `tabs` Map + `tabOrder` array are the single source of truth, and both chrome renderers only ever reflect `tabs:updated` broadcasts.

**The island's expanded states live in a separate always-on-top overlay `WebContentsView`** (`src/renderer/overlay.html`/`overlay.js`, transparent via `setBackgroundColor('#00000000')`), attached to `win.contentView` only while something is showing, so panels float *over* the web content instead of growing the strip and shifting it down. Main owns the mode (`overlayMode`): `'panel'` ‚Äî the command bar expanded in place at the pill's anchor; `'palette'` ‚Äî the same panel summoned by ‚åòL, 120px down over a scrim; `'find'` ‚Äî the find capsule, whose overlay bounds are kept *tight* around the capsule so the rest of the page stays clickable. `showOverlay`/`hideOverlay` in main are the only mutators; the overlay renderer just applies `overlay:show`/`overlay:hide`. Guards that matter: a tab view attached while the overlay is open must be re-stacked below it (re-`addChildView(overlayView)` in `setActiveTab`), Escape dismisses via main's `before-input-event`, and overlay `blur` dismisses panel/palette (not find) ‚Äî but is ignored while a blank-tab focus reclaim is pending, or the reclaim dance re-opens/closes in a loop. The panel's list area shows the tab switcher at rest, slash commands when the input starts with `/`, and the Quick Switcher (loose substring/in-order matching across open tabs, favorites, history) while typing anything else.

**Private tabs** (`/private`, ‚åò‚áßN, `private` on main's tab model): never recorded to history (`history.addVisit`/`updateTitle` guarded per-tab), excluded from session persistence and reopen-closed-tab, inherited by `window.open`/context-menu children, opened to `bowser://newtab/?private=1` (the query flag switches the newtab page's copy + theme). Everything else ‚Äî shield counts, downloads, favorites ‚Äî behaves normally, and the Chromium session (cookies, storage) is deliberately shared. While the active tab is private, both chrome documents set `data-theme="private"` on `<html>`; the pill gets a dashed border, hollow dots, and a "private" chip that closes the tab (quick exit).

**Tab groups** (per the Island Tab Groups exploration in the Bowser Design System): main owns `groups` (`{id, name, collapsed}`, display order = cluster order) alongside `tabs`; each tab carries `groupId` (or null). Groups have **names, not colors** ‚Äî lowercase mono labels. A group exists only while it holds tabs (`pruneEmptyGroups`); creation happens through `/group <name>` (find-or-create) or the inline "new group‚Ä¶" field in the per-row "group" chip's picker, and membership moves via that picker (`‚Üí name` / `‚Üí none`), `/ungroup`, or `/close-group`. The picker's name field survives `tabs:updated` re-renders (value+focus restored across `replaceChildren`) ‚Äî broadcasts arrive ~10/s while any tab loads. The pill renders one dot cluster per group plus a trailing ungrouped cluster (non-active clusters dim; a collapsed non-active group folds into a bordered mini-dot capsule that jumps to the group on click) and shows the active group's name before the domain. The panel's rest list gets per-group headers (click folds/unfolds; collapsed groups show an "N tabs tucked away" row), the Quick Switcher matches group names (ranked above tabs; picking one focuses the group), and ‚åò1‚Äì9 change meaning when groups exist: nth *cluster* (group's first tab, unfolding it) instead of nth tab. New tabs (‚åòT, panel row, `/new`) and `window.open`/context-menu children inherit the active/opener tab's group; new private tabs don't. `session.json` persists `groups` plus a `groupIds` array parallel to `urls`; groups referenced only by private tabs stay out of the file.

**The chrome documents only load their HTML/CSS once, at window creation** (`win.loadFile(...)` + `createOverlay()` in `createMainWindow()`). Editing `src/renderer/index.html`, `overlay.html`, `styles.css`, or either chrome renderer requires relaunching the app to see the change ‚Äî `Cmd/Ctrl+R` reloads the *active tab's* `WebContentsView` (which re-fetches `bowser://.../pages.css` fresh on every navigation), not the chrome. Don't rely on a plain reload to verify a chrome-level change; restart `npm start`.

**`bowser://` internal pages** (`newtab`, `bookmarks`, `history`, `downloads`, `settings`) are served by `src/main/pages.js` over a privileged custom scheme instead of `file://`, so they get a real origin and ordinary web content can never link into local files. The handler resolves every request through `path.basename()` before joining it to `PAGES_DIR` (`src/renderer/pages/`) ‚Äî **it can only serve flat files directly in that one directory, no subdirectories and no `../` traversal.** Any asset (`pages.css`, `icon.svg`, a font file) that `pages/*.html` needs must live flush inside `src/renderer/pages/`. The newtab page is the **"ledger" start page** (direction 1a of the Start Page exploration in the Bowser Design System): date line, "Where to?", favorites, tab groups ("pick up where you left off" ‚Äî clicking one focuses that group), the animated `bowser-sprite` pixel dog (a vendored copy of the design system's `assets/bowser-sprite.js` + its `bowser-sprite-sheet.png` frame sheet ‚Äî a fixed-palette photoreal pixel-art doberman, not recolorable, superseding an earlier hand-drawn/CSS-themed version), and a footer with the weekly blocked count + ‚åòL hint. Live group state and the blocked counter cross from main.js via the `startPage` hooks passed to `setupPages` (`pages:start:data` / `pages:start:focus-group`).

**Security posture:** the chrome window, the overlay view, and every tab's `WebContentsView` run `contextIsolation: true`, `nodeIntegration: false`, `sandbox: true`. Two separate preload scripts, deliberately asymmetric:
- `src/main/preload.js` ‚Äî the rich `browserAPI` bridge (tab lifecycle, window controls, island data + command actions). Attached only to the chrome window and the overlay view.
- `src/main/tab-preload.js` ‚Äî the minimal `bowserPages` bridge, exposed only when `window.location.protocol === 'bowser:'` (re-checked on every navigation, so a tab that leaves an internal page loses it). The main process still independently re-verifies `event.sender.getURL()` on every `pages:*` IPC call in `pages.js` ‚Äî the client-side check is not trusted alone.

**IPC channel namespaces:** `tabs:*` (tab lifecycle, via `browserAPI`), `pages:*` (internal pages' guarded data API, via `bowserPages`), `window:*` (minimize/maximize/close), `chrome:*` (strip/overlay ‚Üí main: layout height, open-island/open-find, and island data + command actions ‚Äî history-list/favorites-list/history-clear/adblock-toggle/adblock-exempt-active/cycle-theme), `overlay:*` (main ‚Üî overlay show/hide/close).

**Address bar focus** is intentionally intricate (`tabsWantingAddressBarFocus`, `reclaimAddressBarFocus()` in `main.js`) because a blank new tab needs the island panel's address input focused rather than its empty page content, and `WebContentsView` focus can settle asynchronously after Electron's own focus/navigation callbacks ‚Äî hence the reassert-on-next-tick pattern rather than a single focus call. `focusAddressBar()` now means "show the overlay panel and focus its input".

**Theming:** CSS custom properties in `:root` (+ a `prefers-color-scheme: dark` override, + an explicit `:root[data-theme="private"]` scope), with the same token names duplicated across `styles.css` (chrome strip *and* overlay ‚Äî one shared stylesheet) and `src/renderer/pages/pages.css` (internal pages) ‚Äî separate files by necessity given the flat-serving constraint above, kept in sync by hand, not shared/imported. `nativeTheme.themeSource` is driven by Settings ‚Üí Appearance (also cycled by `/theme`), so system/light/dark propagates to chrome, internal pages, and web content together, no restart.

**CSP is declared per-file**, not centralized: `index.html`, `overlay.html`, and each of the `pages/*.html` files carry their own `<meta http-equiv="Content-Security-Policy">` tag. Adding any new external resource (a font host, a script source) means updating the CSP in every HTML file that needs it, not one shared config. Fonts currently load from Google Fonts (`fonts.googleapis.com`/`fonts.gstatic.com`) via `<link>` ‚Äî a live network dependency, not bundled locally.

**Ad blocking** (`src/main/adblock.js`): a single `@ghostery/adblocker-electron` engine attached to `session.defaultSession` at startup covers every tab ‚Äî request-level blocking plus cosmetic filtering (the library's own session preload reports DOM state, engine responds with `insertCSS`/`executeJavaScript`). Toggling in Settings calls `enableBlockingInSession`/`disableBlockingInSession` at runtime; blocked-request counts are per-tab and coalesced (~10 broadcasts/s) before hitting the renderer.

**No extension support ‚Äî removed deliberately, don't re-add it.** Bowser used to embed `electron-chrome-extensions` + `electron-chrome-web-store`. The runtime was the app's main source of hard crashes (MV3 service workers tripping Electron's missing `chrome.webRequest` bindings, then faulting inside Chromium ‚Äî V8 traced-reference use-after-free, `SIGSEGV` at `0x130`), it forced the chrome window to run unsandboxed, and it carried a GPL-3.0 licensing constraint. Its only real purpose ‚Äî password managers ‚Äî is unattainable in **any** custom browser shell: Apple's iCloud Passwords helper (`PasswordManagerBrowserExtensionHelper`) and 1Password's native messaging both verify the browser's code signature against an OS/vendor allowlist a custom shell can't join. (The PR to `apple/password-manager-resources` adding Bowser ‚Äî #1137 ‚Äî was approved and merged by @rmondello on 2026-07-04, so Bowser is now in Apple's source data; it still only takes effect if/when Apple ships the updated data in a macOS release, and with the extension runtime gone it's chiefly a legitimacy milestone. In the same thread Apple suggested requesting the `com.apple.developer.web-browser.public-key-credential` entitlement so passkeys/WebAuthn platform-authenticator sign-ins work in Bowser ‚Äî an open roadmap item requiring an Apple Developer capability request plus signing-config changes.) Ad blocking is built in at the network layer instead. On startup `main.js` does a one-time purge of stale extension profile state left by pre-removal versions (`Extensions`, `Extension State`, etc., plus the old `.running` sentinel) ‚Äî the profile's `Service Worker` dir is deliberately left alone, it holds ordinary websites' service workers.

**Naming: "Favorites" vs. `bookmarks`.** The user-facing feature is called **Favorites** (heart icon) per the Bowser Design System ‚Äî but every internal identifier deliberately stays `bookmarks` (`bookmarks.js`, `bookmarks.json`, `pages:bookmarks:*` IPC, the `bowser://bookmarks/` URL, `pages/bookmarks.html`). Change labels/copy only; don't rename the internals, and don't "fix" the mismatch in either direction.

**Persistence** is one `JsonStore` per feature (`src/main/store.js`): a JSON file in `userData`, loaded synchronously once, saved on a 250ms debounce, no schema or migrations. `settings.json`, `bookmarks.json`, `history.json` (capped at 5000 entries), `downloads.json` (capped at 200), `adblock-stats.json` (rolling ads-blocked-this-week counter for the start page; resets lazily at Monday 00:00 local).

**Address input normalization** (`normalizeAddressInput()` in `main.js`) is a deliberately simple regex heuristic ‚Äî has-a-scheme, looks-like-localhost, looks-like-a-domain, else treat as a search query against the engine selected in Settings (DuckDuckGo/Google/Bing/Brave). It will misclassify some edge cases (e.g. paths with dots in query strings); that's a known, accepted limitation, not a bug to silently "fix" with a heavier URL parser unless asked.

**App icon:** `build/icon.png` (1024√ó1024) is the single source electron-builder derives `.icns`/`.ico` from automatically at package time ‚Äî no per-platform icon files to maintain by hand.
