# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

## What this is

Blanc — a minimal Electron browser shell: custom-drawn "Island" chrome (a single floating command pill replacing the traditional tab strip + toolbar, per the Bowser Design System's Island Chrome handoff) with ad/tracker blocking wired in at the network layer, independent of Chrome's extension store and Manifest V3's `declarativeNetRequest` limits. Plus favorites, history, downloads, settings, slash commands, a Quick Switcher, private tabs, an explicit permission policy, and packaging config.

The app was renamed from "Bowser" to Blanc in July 2026 — its former name was inseparable from Nintendo's Mario villain, which didn't fit a serious indie browser's ambitions. `build.appId` in `package.json` deliberately stayed `me.bnfy.bowser` through the rename (only `name`/`productName` changed) to preserve macOS Gatekeeper/notarization identity and the auto-update chain for existing installs — don't "clean up" that mismatch, it's intentional. The design system's project name ("Bowser Design System", referenced throughout this file) is a separate Codex Design project the user maintains and wasn't renamed alongside the app; leave those references as-is.

## Commands

```
npm install
npm start                       # run in dev (electron .)
npm run dist                    # installable build: macOS dmg/zip, Windows NSIS, Linux AppImage
npm run dist:dir                # quick unpacked build in dist/, no installer
npm run release                 # scripts/release.sh: clean, build, sign, publish to GitHub Releases
```

There is no test suite and no linter configured in this repo — don't assume `npm test` or `npm run lint` exist.

`npm run release` bumps nothing itself — bump `version` in `package.json` (and consider the `electron` devDependency, since Chromium can't be swapped out of a running app) *before* running it. It shells out to `scripts/release.sh`, which authenticates via the `gh` CLI's own cached session (no `GH_TOKEN` needed locally), builds unpublished, then uses `gh release create` directly instead of electron-builder's own GitHub publisher — that publisher races its per-artifact upload tasks against each other on first publish for a tag and can leave a release missing `latest-mac.yml` or a blockmap; creating the release once with the complete macOS asset set avoids that. **Released versions are immutable:** the script checks local/remote tags and GitHub releases before building and fails if the version already exists; always bump to a new version, never overwrite assets. It also refuses dirty desktop release sources and wipes `dist/` before building so artifacts correspond to committed code and stale files never linger.

**Notarization:** the build step is wrapped in `op run --env-file=.env.1password --no-masking -- ...`, pulling `APPLE_ID`/`APPLE_APP_SPECIFIC_PASSWORD` from the 1Password item **"Apple Notarization"** (vault Dev, `username`/`password` fields) plus a literal `APPLE_TEAM_ID` — electron-builder's `@electron/notarize` integration auto-detects those three env vars with no extra config (hardened runtime + the JIT/unsigned-memory/disable-library-validation entitlements it needs are already electron-builder's defaults). If `op` isn't installed, the script falls back to an unnotarized build with a warning instead of failing outright; if `op` *is* installed but the item can't be resolved (locked vault, wrong item name), it fails loudly rather than silently shipping unnotarized — a signed-but-unnotarized build gets quarantined and blocked by Gatekeeper the moment it's downloaded (e.g. by `electron-updater` during an auto-update), which is exactly what broke a stale pre-rebrand install after the v0.3.0 release.

**Windows & Linux:** `release.sh` only builds macOS locally (it's a macOS dev machine), then dispatches `.github/workflows/release-windows-linux.yml` (`gh workflow run ... -f tag=$TAG`) and watches it to completion. That workflow builds NSIS on `windows-latest` and AppImage on `ubuntu-latest` and uploads both onto the same new release tag; duplicate asset uploads fail rather than overwrite an existing version. `workflow_dispatch` only works off a workflow file already on the default branch, so a brand-new/edited workflow won't be dispatchable until merged — `release.sh` warns and exits 0 (without failing the mac release) if the dispatch call fails, printing the manual `gh workflow run` command to run once that's sorted. Windows code signing is opportunistic, tried in order: **Azure Trusted Signing** first (repo secrets `AZURE_TENANT_ID`/`AZURE_CLIENT_ID`/`AZURE_CLIENT_SECRET` for a service principal with the "Trusted Signing Certificate Profile Signer" role, plus repo *variables* — non-secret — `AZURE_TRUSTED_SIGNING_ENDPOINT`/`AZURE_CODE_SIGNING_ACCOUNT_NAME`/`AZURE_CERTIFICATE_PROFILE_NAME`/`AZURE_PUBLISHER_NAME` identifying the signing account/profile, passed to electron-builder via `--config.win.azureSignOptions.*` CLI overrides rather than baked into `package.json` since they're account-specific and may not be finalized); then a traditional Authenticode cert via `CSC_LINK`/`CSC_KEY_PASSWORD` secrets (electron-builder's own auto-detected env vars) if Azure isn't configured; otherwise it builds unsigned with a `::warning::` annotation. An unsigned installer triggers a SmartScreen "unknown publisher" prompt, and even a traditionally-signed OV cert builds SmartScreen reputation gradually per binary hash — Azure Trusted Signing is the one that gets immediate trust, which is why it's tried first. Linux AppImages aren't code-signed (no OS-level equivalent to Gatekeeper/SmartScreen to satisfy).

The Azure branch in that workflow requires **all three** of `AZURE_CLIENT_ID`, `AZURE_CODE_SIGNING_ACCOUNT_NAME`, and `AZURE_CERTIFICATE_PROFILE_NAME` to be non-empty before it's attempted — a partially-configured Azure setup (e.g. the signing account created but the certificate profile still stuck in identity validation) must fall through to `CSC_LINK`/unsigned rather than attempt Azure with an empty `certificateProfileName`, which fails hard (`Invoke-TrustedSigning: Cannot bind argument ... because it is an empty string`) instead of degrading gracefully — this exact failure mode shipped and broke a real release before the gate was tightened. Two electron-builder artifact-naming quirks the workflow's verify/upload steps depend on, found by reading actual CI output rather than the docs: the AppImage gets **no** `-x86_64`/arch suffix when built for the runner's default arch (only a non-default or multi-arch build adds one), and AppImage's block map is embedded in the file itself rather than shipped as a separate `.blockmap` (unlike the mac zip/dmg and the NSIS `.exe`, which do get one) — re-check the real `dist/` output rather than assuming a pattern if this ever needs to change. v0.9.6 was the first release with all three platforms live; macOS and Windows were confirmed installing on real hardware, Linux via the CI build/upload succeeding (x86_64 AppImages can't be verified locally on Apple Silicon without a slow, non-default emulated VM, since Parallels' native Linux VMs on Apple Silicon are ARM64-only).

First launch is slower than usual: the ad blocker fetches and compiles EasyList + EasyPrivacy, then caches the compiled engine in the app's userData directory (`adblock-engine.v<N>.bin`) so later launches are instant. Delete that file to force a refresh.

Auto-update (`electron-updater` against GitHub Releases) only runs in packaged builds — `npm start` skips it entirely. Releasing means bumping the `electron` devDependency (it tracks Chromium stable) and `version` in `package.json`, since Chromium itself can't be swapped out of a running app.

## Architecture

**One `BrowserWindow`, many `WebContentsView`s.** The window's own `webContents` renders `src/renderer/index.html`/`styles.css` — the chrome **strip**: a slim (64px) band holding the resting island pill (back/forward, the active group's tab dots with a capped overflow, favicon, domain, shield count, private chip, and a trailing reload/favorite/downloads action cluster), window controls, and permission prompts. The whole resting pill is scaled up in one place with `zoom` on `#islandPill` (kept in lockstep with the `--strip-h` geometry token); each anonymous tab dot blooms into a disc showing that tab's favicon on hover/focus (`.dot-peek`, reusing the pill's favicon rendering) so you can tell which tab you're switching to before you click. Each tab is a separate `WebContentsView` added as a child of `win.contentView` at `y = chromeHeight`; only the active tab's view is attached, so switching tabs is remove-one/add-another rather than destroying anything. `src/main/main.js` owns all of this: the `tabs` Map + `tabOrder` array are the single source of truth, and both chrome renderers only ever reflect `tabs:updated` broadcasts.

**The island's expanded states live in a separate always-on-top overlay `WebContentsView`** (`src/renderer/overlay.html`/`overlay.js`, transparent via `setBackgroundColor('#00000000')`), attached to `win.contentView` only while something is showing, so panels float *over* the web content instead of growing the strip and shifting it down. Main owns the mode (`overlayMode`): `'panel'` — the command bar expanded in place at the pill's anchor; `'palette'` — the same panel summoned by ⌘L, 120px down over a scrim; `'find'` — the find capsule, whose overlay bounds are kept *tight* around the capsule so the rest of the page stays clickable. `showOverlay`/`hideOverlay` in main are the only mutators; the overlay renderer just applies `overlay:show`/`overlay:hide`. Guards that matter: a tab view attached while the overlay is open must be re-stacked below it (re-`addChildView(overlayView)` in `setActiveTab`), Escape dismisses via main's `before-input-event`, and overlay `blur` dismisses panel/palette (not find) — but is ignored while a blank-tab focus reclaim is pending, or the reclaim dance re-opens/closes in a loop. The panel's list area shows the tab switcher at rest, slash commands when the input starts with `/`, and the Quick Switcher (loose substring/in-order matching across open tabs, favorites, history) while typing anything else.

**Private tabs** (`/private`, ⌘⇧N, `private` on main's tab model): never recorded to history (`history.addVisit`/`updateTitle` guarded per-tab), excluded from session persistence and reopen-closed-tab, inherited by `window.open`/context-menu children, opened to `blanc://newtab/?private=1` (the query flag switches the newtab page's copy + theme). Everything else — shield counts, downloads, favorites — behaves normally, and the Chromium session (cookies, storage) is deliberately shared. While the active tab is private, both chrome documents set `data-theme="private"` on `<html>`; the pill gets a dashed border, hollow dots, and a "private" chip that closes the tab (quick exit).

**Tab groups** (per the Island Tab Groups exploration in the Bowser Design System): main owns `groups` (`{id, name, collapsed}`, display order = cluster order) alongside `tabs`; each tab carries `groupId` (or null). Groups have **names, not colors** — lowercase mono labels. A group exists only while it holds tabs (`pruneEmptyGroups`); creation happens through `/group <name>` (find-or-create) or the inline "new group…" field in the per-row "group" chip's picker, and membership moves via that picker (`→ name` / `→ none`), `/ungroup`, or `/close-group`. The picker's name field survives `tabs:updated` re-renders (value+focus restored across `replaceChildren`) — broadcasts arrive ~10/s while any tab loads. The pill now renders **only the active tab's group** as dots (capped at 8, with a quiet `+N` that opens the panel) and shows the active group's name before the domain. Pinning is group-local: a grouped pin stays in that group and leads its rows/dots, while only ungrouped pins use the standalone pinned section in the ⌘L panel. Other groups also live in that panel behind per-group headers (click folds/unfolds; collapsed groups show an "N tabs tucked away" row). The Quick Switcher matches group names (ranked above tabs; picking one focuses the group), and ⌘1–9 change meaning when groups exist: nth *cluster* (group's first tab, unfolding it) instead of nth tab. A plain new tab (⌘T, panel row, `/new`) always launches ungrouped, regardless of the active tab's group — join a group afterward via the row's "group" chip picker. `window.open`/context-menu children are the exception: they still inherit the opener tab's group, since a link opened from a grouped tab is presumed to belong with it. New private tabs are never grouped. `session.json` persists `groups` plus a `groupIds` array parallel to `urls`; groups referenced only by private tabs stay out of the file.

**The chrome documents only load their HTML/CSS once, at window creation** (`win.loadFile(...)` + `createOverlay()` in `createMainWindow()`). Editing `src/renderer/index.html`, `overlay.html`, `styles.css`, or either chrome renderer requires relaunching the app to see the change — `Cmd/Ctrl+R` reloads the *active tab's* `WebContentsView` (which re-fetches `blanc://.../pages.css` fresh on every navigation), not the chrome. Don't rely on a plain reload to verify a chrome-level change; restart `npm start`.

**`blanc://` internal pages** (`newtab`, `bookmarks`, `history`, `downloads`, `settings`) are served by `src/main/pages.js` over a privileged custom scheme instead of `file://`, so they get a real origin and ordinary web content can never link into local files. The handler resolves every request through `path.basename()` before joining it to `PAGES_DIR` (`src/renderer/pages/`) — **it can only serve flat files directly in that one directory, no subdirectories and no `../` traversal.** Any asset (`pages.css`, `icon.svg`, a font file) that `pages/*.html` needs must live flush inside `src/renderer/pages/`. The newtab page is the **"ledger" start page** (direction 1a of the Start Page exploration in the Bowser Design System): date line, "Where to?", favorites, tab groups ("pick up where you left off" — clicking one focuses that group), and a footer with the weekly blocked count + ⌘L hint. There's deliberately no mascot on this page — an earlier version had an animated pixel-art dog sprite here, retired along with the "Bowser" name as part of the July 2026 rebrand; don't reintroduce one without being asked. Live group state and the blocked counter cross from main.js via the `startPage` hooks passed to `setupPages` (`pages:start:data` / `pages:start:focus-group`).

**Security posture:** the chrome window, the overlay view, and every tab's `WebContentsView` run `contextIsolation: true`, `nodeIntegration: false`, `sandbox: true`. Three separate preload scripts, deliberately asymmetric:
- `src/main/preload.js` — the rich `browserAPI` bridge (tab lifecycle, window controls, island data + command actions). Attached only to the chrome window and the overlay view.
- `src/main/tab-preload.js` — the minimal `bowserPages` bridge, exposed only when `window.location.protocol === 'blanc:'` (re-checked on every navigation, so a tab that leaves an internal page loses it). The main process still independently re-verifies `event.sender.getURL()` on every `pages:*` IPC call in `pages.js` — the client-side check is not trusted alone. `bowserPages`/`bowserAuth` (the analogous bridge for the basic-auth dialog) kept their names across the rebrand — they're internal-only identifiers, same rule as the Favorites/`bookmarks` split below.
- `src/main/chrome-compat-preload.js` — an unprivileged session-wide `window.chrome.app`/`csi`/`loadTimes` compatibility shim. It is registered through `session.registerPreloadScript` so both ordinary tabs and Chromium-adopted OAuth children receive it without replacing the child's `webPreferences.preload` (which severs `window.opener`). It exposes no IPC or application data.

**IPC channel namespaces:** `tabs:*` (tab lifecycle, via `browserAPI`), `pages:*` (internal pages' guarded data API, via `bowserPages`), `window:*` (minimize/maximize/close), `chrome:*` (strip/overlay → main: layout height, open-island/open-find, and island data + command actions — history-list/favorites-list/history-clear/adblock-toggle/adblock-exempt-active/cycle-theme), `overlay:*` (main ↔ overlay show/hide/close).

**Address bar focus** is intentionally intricate (`tabsWantingAddressBarFocus`, `reclaimAddressBarFocus()` in `main.js`) because a blank new tab needs the island panel's address input focused rather than its empty page content, and `WebContentsView` focus can settle asynchronously after Electron's own focus/navigation callbacks — hence the reassert-on-next-tick pattern rather than a single focus call. `focusAddressBar()` now means "show the overlay panel and focus its input".

**Theming:** CSS custom properties in `:root` (+ a `prefers-color-scheme: dark` override, + an explicit `:root[data-theme="private"]` scope), with the same token names duplicated across `styles.css` (chrome strip *and* overlay — one shared stylesheet) and `src/renderer/pages/pages.css` (internal pages) — separate files by necessity given the flat-serving constraint above, kept in sync by hand, not shared/imported. `nativeTheme.themeSource` is driven by Settings → Appearance (also cycled by `/theme`), so system/light/dark propagates to chrome, internal pages, and web content together, no restart.

**CSP is declared per-file**, not centralized: `index.html`, `overlay.html`, and each of the `pages/*.html` files carry their own `<meta http-equiv="Content-Security-Policy">` tag. Adding any new external resource (a font host, a script source) means updating the CSP in every HTML file that needs it, not one shared config. Fonts currently load from Google Fonts (`fonts.googleapis.com`/`fonts.gstatic.com`) via `<link>` — a live network dependency, not bundled locally.

**Ad blocking** (`src/main/adblock.js`): a single `@ghostery/adblocker-electron` engine attached to `session.defaultSession` at startup covers every tab — request-level blocking plus cosmetic filtering (the library's own session preload reports DOM state, engine responds with `insertCSS`/`executeJavaScript`). Toggling in Settings calls `enableBlockingInSession`/`disableBlockingInSession` at runtime; blocked-request counts are per-tab and coalesced (~10 broadcasts/s) before hitting the renderer.

**No extension support — removed deliberately, don't re-add it.** Blanc (then still named Bowser) used to embed `electron-chrome-extensions` + `electron-chrome-web-store`. The runtime was the app's main source of hard crashes (MV3 service workers tripping Electron's missing `chrome.webRequest` bindings, then faulting inside Chromium — V8 traced-reference use-after-free, `SIGSEGV` at `0x130`), it forced the chrome window to run unsandboxed, and it carried a GPL-3.0 licensing constraint. Its only real purpose — password managers — is unattainable in **any** custom browser shell: Apple's iCloud Passwords helper (`PasswordManagerBrowserExtensionHelper`) and 1Password's native messaging both verify the browser's code signature against an OS/vendor allowlist a custom shell can't join. (The PR to `apple/password-manager-resources` adding Bowser — #1137 — was approved and merged by @rmondello on 2026-07-04, so the app is now in Apple's source data under its former name; it predates the rename to Blanc, and a follow-up PR under the new name is a separate later task. It still only takes effect if/when Apple ships the updated data in a macOS release, and with the extension runtime gone it's chiefly a legitimacy milestone. Electron's Touch ID WebAuthn API now lets Blanc create and use device-bound Secure Enclave passkeys in its own keychain access group; `src/main/webauthn.js` and the macOS entitlement plists must keep that group in sync. It does **not** read existing iCloud/third-party credential-manager passkeys. Apple's `com.apple.developer.web-browser.public-key-credential` entitlement remains the open path for those credentials.) Ad blocking is built in at the network layer instead. On startup `main.js` does a one-time purge of stale extension profile state left by pre-removal versions (`Extensions`, `Extension State`, etc., plus the old `.running` sentinel) — the profile's `Service Worker` dir is deliberately left alone, it holds ordinary websites' service workers.

**Naming: "Favorites" vs. `bookmarks`.** The user-facing feature is called **Favorites** (heart icon) per the Bowser Design System — but every internal identifier deliberately stays `bookmarks` (`bookmarks.js`, `bookmarks.json`, `pages:bookmarks:*` IPC, the `blanc://bookmarks/` URL, `pages/bookmarks.html`). Change labels/copy only; don't rename the internals, and don't "fix" the mismatch in either direction.

**Persistence** is one `JsonStore` per feature (`src/main/store.js`): a JSON file in `userData`, loaded synchronously once, saved on a 250ms debounce, no schema or migrations. `settings.json`, `bookmarks.json`, `history.json` (capped at 5000 entries), `downloads.json` (capped at 200), `adblock-stats.json` (rolling ads-blocked-this-week counter for the start page; resets lazily at Monday 00:00 local). Because `userData`'s path is derived from `productName` (not `appId`, which stayed stable across the rebrand — see "What this is" above), `main.js` runs a one-time startup migration for packaged builds: if the new `Blanc`-named userData directory doesn't exist yet but the old `Bowser`-named one does, it's copied forward wholesale before anything else touches it. This is a permanent, one-time-per-install shim, not a bug — existing users' history/favorites/settings survive the rename transparently. Don't remove it casually; it's still doing real work for anyone who hasn't relaunched since upgrading past the rename.

**Telemetry** (`src/main/telemetry.js`): a single launch ping — **on by default, opt-out** (Settings → "Help improve Blanc", `usagePing`; the default flipped from off→on when active-user metrics landed, so `usagePing`'s default is `true` in both `settings.js` and `settings-schema/schema.json`), packaged builds only (`app.isPackaged` guard so dev runs don't inflate counts), fire-and-forget (a failed or blocked ping must never affect startup or surface to the user). Payload is `{installId, sessionId, version, platform, arch}` — `sessionId` is a random positive 32-bit integer per launch, used as GA4's `session_id` so the Measurement Protocol counts sessions. `installId` is a random per-install token minted once and kept in its **own** device-local store `install.json` (deliberately NOT in settings, so it never crosses Profile Sync, and clearing it is a single file delete) — it maps to a device install, never a person: no name, account, IP, or browsing data rides with it. It POSTs to the `blanc-ping` Cloudflare Worker whose source lives in `cloudflare/ping-worker/`. The worker counts two things from one ping: **launches** (aggregate `total`/`day:`/`version:`/`platform:` counters — repeat launches all count) and **active users** (the first ping from an install in a given day/week/month bumps that period's unique counter via a TTL'd `seen:*` flag, so DAU/WAU/MAU and month-over-month retention are deduped). Bearer-token-gated `GET /stats` returns both, plus retention; it optionally mirrors to GA4 (using `installId` as GA `client_id`) when the `GA_API_SECRET` worker secret is set. The install id is a genuine persistent identifier — a deliberate, minimal departure from the app's otherwise identifier-free stance, made to count distinct users; keep it opaque, keep it out of sync, and don't attach anything to it. If the worker URL ever changes, update `PING_ENDPOINT` in `telemetry.js`.

**Blanc Supporter** (`src/main/supporter.js`): a $19 one-time Polar.sh license that unlocks three supporter-only Dock colorways (`ember`, `plum`, `gold` — generated by `scripts/gen-supporter-icons.sh` from `icon-default.png`, same fixed geometry). Activation happens once in the main process against Polar's customer-portal API (dev builds hit the sandbox; `POLAR_ORGANIZATION_ID` in `supporter.js` is the production org's id, hardcoded) and writes `settings.supporter` via `setSupporter()` — the only writer; the generic `setSettings()` whitelist deliberately ignores it. Once set, the record is trusted forever: no revalidation, no phone-home, works offline — perks are cosmetics, DRM would betray the brand. Renderers only ever see a derived `supporterActive` boolean, never the key. `settings.js` exports one `isAppIconAllowed()` predicate that both `setSettings()` (write) and `getSettings()` (read) apply — a stale/unlicensed supporter icon id (hand-edited or copied settings.json) reads back as the default (`paper`) everywhere a renderer or `applyAppIcon()` sees it, not just at the Dock. Setup runbook: `docs/polar-setup.md`.

**Profile Sync** (`src/main/sync.js` + `src/main/sync-crypto.js`): opt-in, end-to-end-encrypted cross-device sync of **Favorites and settings** (off by default, local-first). A user sets a sync name + passphrase once (Settings → Sync); `sync-crypto.js` derives an opaque `accountId` and an AES-256-GCM key via `scrypt`→`HKDF` (Node built-ins, no dependency), the passphrase is then discarded and only the derived key persists in `sync.json`. Each store's plaintext is merged client-side (`bookmarks.exportForSync`/`mergeFromSync`, ditto `settings`) and pushed as ciphertext to the `blanc-sync` Cloudflare Worker (`cloudflare/sync-worker/`), which stores only opaque blobs keyed by `accountId` — it cannot read, index, or merge anything. Merges are union-based (Favorites keyed by url + delete tombstones; settings per-key LWW via a `_syncMeta` clock) so a sync never silently drops data. **Never synced:** `supporter` (license-sharing vector), `appIcon` (device-local), `usagePing` (per-install consent), cookies/storage, and — in v1 — history/downloads/permissions. All crypto and `net.fetch` are main-process only; the renderer sends the passphrase once over the guarded `pages:settings:sync-enable` IPC and otherwise sees status-only responses (no keys). The stored derived key is no weaker than the app's already-plaintext local stores — the guarantee is server-blindness, not local-at-rest encryption. Worker deploy/runbook: `cloudflare/sync-worker/README.md`.

**Address input normalization** (`normalizeAddressInput()` in `main.js`) is a deliberately simple regex heuristic — has-a-scheme, looks-like-localhost, looks-like-a-domain, else treat as a search query against the engine selected in Settings (DuckDuckGo/Google/Bing/Brave). It will misclassify some edge cases (e.g. paths with dots in query strings); that's a known, accepted limitation, not a bug to silently "fix" with a heavier URL parser unless asked. `handOffToOs()` (also in `main.js`) is checked ahead of it wherever a URL becomes a navigation target — page-initiated navigation, `window.open` children, the context menu's "Open Link" actions, and the address bar's raw typed text before `normalizeAddressInput()` ever sees it — so a bare `mailto:`/`tel:`/`facetime:`/`sms:` URI (which has no `://` and would otherwise be misread as a search query or a bogus domain) is handed to the OS instead.

**App icon:** `build/icon.png` (1024×1024) is the single source electron-builder derives `.icns`/`.ico` from automatically at package time — no per-platform icon files to maintain by hand. It and the eleven Dock colorway PNGs in `src/renderer/pages/` all share one fixed layout — an 824×824 rounded square inset 100px into the 1024 canvas, with the mark scaled to a 522px-tall bounding box dead-centered — so a future logo change only needs the mark re-rendered at that same geometry per colorway, not the layout re-derived from scratch. Eight are free and picked directly in Settings (`icon-{paper,ink,graphite,default,midnight,cream,forest,sage}.png` — labels live in `settings.js`'s `APP_ICON_LABELS`, `paper` is the current default, `default` is the original green colorway whose id is frozen for saved settings); three are supporter-gated (`icon-{ember,plum,gold}.png`, see Blanc Supporter above).

## Cross-platform parity infrastructure (`spec/`, `tokens/`, `settings-schema/`, `copy/`, `test/desktop/`)

Scaffolding for a **future** native iOS/Android port (landed July 2026, PR #13). The mobile apps don't exist yet — this Electron app is the **reference implementation**, and everything here is additive to it except one test-only, env-gated hook.

**`spec/` is the platform-neutral source of truth for what Blanc *is*.** Features `F1–F24` define the user-facing contract each platform must implement; the divergence register `D1–D14` documents where platforms deliberately differ (e.g. the ad-blocking engine — desktop/Android programmatic vs. iOS's declarative `WKContentRuleList`, which reintroduces a Manifest-V3-style rule cap) plus the parity contract that still holds across the split; `blocking-backends.md` specs the two mobile blocking engines concretely; `parity-matrix.md` is the dashboard; `spec/acceptance/*.feature` are shared Gherkin scenarios. The governing rule: **parity means product/behaviour parity, not implementation parity** — a divergence is *documented* (add a `D#`), not avoided. `spec/README.md` is the entry point. It's a spec, not code; editing it doesn't change the app.

**Three "single source of truth" substrates generate the (future) mobile code *and* guard the desktop against drift.** `tokens/tokens.json` (design tokens), `settings-schema/schema.json` (settings enums + defaults), and `copy/slash-commands.json` (slash-command copy) each have a `build.mjs`: `npm run <name>:build` emits the mobile artifacts under `*/generated/` (Swift/Kotlin, plus iOS `.strings` / Android `.xml`), and `npm run <name>:check` verifies the **desktop** hasn't drifted from that source. `npm run substrate:check` runs all three, plus `adblock:check` — a same-shaped freshness guard that verifies the committed `WKContentRuleList` blocklist (`adblock/generated/`) still matches the pinned EasyList/EasyPrivacy sources (`adblock/` isn't a mobile-codegen substrate, just a generated artifact kept honest the same way); `.github/workflows/parity-guards.yml` runs it (+ the acceptance dry-run) on every push/PR, so drift fails CI.

**The critical operational rule:** the desktop CSS/JS is *guarded, not generated* (a headless build can't visually re-verify chrome), so the source-of-truth JSON and the desktop file are kept in lockstep by the check, not by codegen. If you change a token value in `styles.css`/`pages.css` (the Theming section's `:root` custom properties), a default/enum in `settings.js` (search engines, themes, app icons — Blanc Supporter / App icon sections), or a slash-command hint in `overlay.js`/`pages/shortcuts.js`, you **must** also update the matching JSON and run `<name>:build`, or `substrate:check` fails — and vice versa. Never hand-edit `*/generated/` files. The checkers parse those desktop files with comment-stripping + line-anchored regexes to avoid false-negatives; if you substantially restructure them (single-line vs. one-per-line declarations), re-verify with `node <name>/build.mjs --check`.

**`test/desktop/` is a desktop acceptance harness** (Cucumber + Playwright-Electron) binding the `spec/acceptance/` scenarios to desktop step definitions (24 executing; the rest a tagged backlog). It drives the app through **`src/main/test-hook.js`, a test-only main-process surface `main.js` installs only when `!app.isPackaged && process.env.BLANC_TEST === '1'`** — airtight (never a packaged build; `BLANC_TEST=0/false` stays off). That env-gate is the *only* parity change to shipping code; in test mode it skips the network ad-engine build so the app launches offline and exposes state on `globalThis.__blanc`. `npm run test:acceptance:dry` resolves step definitions without launching; `npm run test:acceptance:desktop` runs them (prefix `xvfb-run -a` on headless Linux; needs the Electron binary).
